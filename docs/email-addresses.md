# Email addresses referenced in the codebase

Every address on a domain we control, where it comes from, and whether it
needs to exist in Google Workspace. Generated by sweeping the repo for
anything matching an email pattern (146 hits; the ~130 `@example.com` ones are
test fixtures and are excluded here).

## Must exist — user-facing

Each of these is shown to a real person as a way to contact us. If the mailbox
doesn't exist, the reply bounces and the person concludes we ignored them.

| Address | Purpose | Where it's surfaced |
|---------|---------|---------------------|
| `support@trusthalal.org` | General support | Mobile Profile → Contact support; owner portal (3 pages); the `contact` block in the API's OpenAPI spec (`api/app/main.py`); App Store listing copy |
| `hello@trusthalal.org` | Front-door / sales | Brand site (`apps/brand/src/lib/links.ts`); all owner + consumer + community marketing copy |
| `verifiers@trusthalal.org` | Verifier programme | Consumer site: `become-a-verifier` page and `verifier-gate` component |
| `ethics@trusthalal.org` | AI-ethics feedback | `content/ethics/ai-ethics.md`, brand site links |
| `privacy@trusthalal.org` | Privacy / data requests | Privacy policy, brand site links |
| `legal@trusthalal.org` | Formal legal notices, DMCA | Terms of service (`apps/brand/src/app/terms/page.tsx`), brand site links |

`privacy@` matters more than its single reference suggests: the published
privacy policy names it as the route for GDPR and CCPA requests, which are
legally time-boxed. A bounce there is the one on this list with a deadline
attached.

`legal@` is now a second one with a deadline attached, and it is the newer
risk. The terms of service publish it as the address for DMCA takedown
notices and counter-notices, and DMCA safe harbour depends on notices
actually being received and acted on. An unmonitored `legal@` is worse than
publishing no address at all — it advertises a channel we are ignoring.
**Create this alias before the terms page goes live.**

## Must exist — machine

| Address | Purpose | Where it's set |
|---------|---------|----------------|
| `noreply@trusthalal.org` | `From:` on every transactional email | `RESEND_FROM_EMAIL`, default `"Trust Halal <noreply@trusthalal.org>"` |
| `appreview@trusthalal.org` | App Review demo account login | `api/scripts/create_review_demo_account.py` |

### `noreply@` has a real gap

`RESEND_REPLY_TO` is unset (`api/.env.example` line 63, `config.py` default
`None`), and the email templates carry **no contact address at all** — the
footer is the wordmark, a tagline and two URLs. So the only thing a recipient
can do with a claim decision, a dispute outcome, a password reset or a review
takedown notice is hit Reply, which goes to `noreply@trusthalal.org` and
disappears. That's exactly the failure this audit was for.

Two ways to close it, and they're not mutually exclusive:

1. Set `RESEND_REPLY_TO=support@trusthalal.org` in the API environment. One
   variable, no deploy of code, and replies land somewhere staffed.
2. Create `noreply@` as an alias forwarding to `support@` — belt and braces,
   and it also catches anyone who mails it directly.

Worth doing (1) regardless: it's the only one that changes what the recipient's
mail client actually does.

### `appreview@` needs to receive mail even though nobody reads it

The demo account is created pre-verified, so it never needs a confirmation
email — but it's a normal user account, so the platform will send it
transactional mail (a review of theirs being moderated, for instance). Those
would hard-bounce against a nonexistent mailbox on our own domain, and
repeated hard bounces are what damages sender reputation with Resend. An alias
pointing at `support@` costs nothing and avoids that.

## Do NOT create

| Address | Why not |
|---------|---------|
| `admin@`, `owner@`, `consumer@`, `verifier@` `trusthalal.dev` | Local dev seed accounts (`api/scripts/seed_dev.py`, `api/README.md`). Different domain, never receives mail. |
| `amir@amirs.co`, `contact@amirs.co` | Fictional restaurant in a design mockup (`docs/2026-07-15-owner-onboarding-wizard.html`). |
| `contact@yourrestaurant.com` | Placeholder text in owner-portal form inputs — it's the greyed-out example, not a real address. |
| `rozebm@gmail.com` | Package author metadata in `api/pyproject.toml` and an example in a script's docstring. |
| ~130 `@example.com` addresses | Test fixtures. `example.com` is IANA-reserved for exactly this. |

## A drift note

`apps/brand/src/lib/links.ts` exports four of these as constants, but the other
apps hardcode their own copies: `support@` appears literally in three owner
pages, one mobile screen and `api/app/main.py`; `verifiers@` in two consumer
files. Nothing is wrong today — every copy agrees — but changing an address
means grepping rather than editing one file. Worth knowing before you rename
one.
